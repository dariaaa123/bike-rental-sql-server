Platformă de închiriere biciclete – Documentație BD (SQL Server / SSMS)

Proiect: Bază de date relațională (8 tabele + relație m-n), proceduri, triggere, job, drepturi, backup/restore.

1. Specificații proiect

  • SGBD Microsoft SQL Server (SSMS)
  • Nr. tabele 8 tabele principale + 1 tabel de legătură pentru relația m-n
  • Relații 1-n și m-n
  • Integritate PK, FK, CHECK, UNIQUE
  • Indexuri Clustered (PK implicit) + Nonclustered pe coloane uzuale
  • Views rapoarte (disponibilitate, venituri, top clienți)
  • Stored Procedures CRUD + Start/End împrumut
  • Triggere DML (consistență business) + DDL (audit)
  • Cursor raport disponibilitate pe stații
  • Securitate roluri de DB + permisiuni
  • Job reconciliere status biciclete (SQL Server Agent)
  • Backup strategie + scripturi exemplu

2. Domeniu & descriere

Sistemul gestionează biciclete disponibile în stații, închirieri (împrumuturi) efectuate de clienți și plăți asociate. Include și funcționalitatea de stații favorite (relație m-n între clienți și stații).

3. Cazuri de utilizare

3.1 Actori

  • Client – închiriază bicicletă, închide împrumut, setează stații favorite
  • Operator – administrează biciclete/stații, inițiază/închide împrumuturi
  • Casier – înregistrează plăți și consultă rapoarte de venituri
  • Admin aplicație – administrare completă obiecte BD (în cadrul DB)

3.2 Listă cazuri de utilizare

  • UC1: Administrare tipuri biciclete (CRUD)
  • UC2: Administrare stații (CRUD)
  • UC3: Administrare biciclete (CRUD + status)
  • UC4: Înregistrare client (CRUD)
  • UC5: Start împrumut (validare bicicletă disponibilă)
  • UC6: End împrumut (închidere + returnare stație)
  • UC7: Înregistrare plată
  • UC8: Setare stații favorite (m-n)
  • UC9: Rapoarte (views)
  • UC10: Audit modificări schemă (DDL trigger)

4. Structura bazei de date

4.1 Tabele

Tabel Rol / descriere Chei & relații (rezumat)
TipuriBiciclete Catalog tipuri (oraș, mountain, electric etc.) PK: TipID; UNIQUE: NumeTip; 1-n cu Biciclete
Statii Stații de preluare/returnare PK: StatieID; 1-n cu Biciclete; m-n cu Clienti prin ClientiStatiiFavorite
Biciclete Inventar biciclete + status PK: BicicletaID; FK: TipID, StatieID; CHECK Status
Clienti Clienți care pot închiria biciclete PK: ClientID; UNIQUE: Email; 1-n cu Imprumuturi
Imprumuturi Istoric închirieri (active/închise) PK: ImprumutID; FK: ClientID, BicicletaID; CHECK dată
Plati Plăți asociate împrumuturilor PK: PlataID; FK: ImprumutID; CHECK Suma>0; DEFAULT DataPlata
Roluri Roluri de aplicație (stocate în BD) PK: RolID; UNIQUE: NumeRol; 1-n cu Utilizatori
Utilizatori Conturi aplicație (hash parola + rol) PK: UtilizatorID; UNIQUE: Username; FK: RolID
ClientiStatiiFavorite m-n Tabel de legătură: clienți ↔ stații favorite PK compus: (ClientID, StatieID); FK către Clienti și Statii

4.2 Relații

  • 1-n: TipuriBiciclete → Biciclete
  • 1-n: Statii → Biciclete
  • 1-n: Clienti → Imprumuturi
  • 1-n: Biciclete → Imprumuturi
  • 1-n: Imprumuturi → Plati
  • 1-n: Roluri → Utilizatori
  • m-n: Clienti ↔ Statii prin ClientiStatiiFavorite

5. Constrângeri & indexuri

5.1 Constrângeri

  • PK pe fiecare tabel (clustered implicit în SQL Server)
  • FK între tabelele relaționate pentru integritate referențială
  • UNIQUE:
    • Clienti.Email
    • Utilizatori.Username
    • Roluri.NumeRol
    • TipuriBiciclete.NumeTip
  • CHECK:
    • Biciclete.Status ∈ {'disponibila','inchiriata','mentenanta'}
    • Plati.Suma > 0
    • Imprumuturi: DataSfarsit este NULL sau ≥ DataInceput

5.2 Indexuri

  • Clustered: cheile primare (implicit)
  • Nonclustered:
    • IX_Biciclete_StatieID (căutare biciclete pe stație)
    • IX_Biciclete_Status (căutare biciclete pe status)
    • IX_Imprumuturi_ClientID (istoric împrumuturi per client)
    • IX_Plati_ImprumutID (plăți per împrumut)

6. Vederi (Views) – descriere

  • vBicicleteDisponibilePeStatie – număr biciclete disponibile pe stație și tip
  • vImprumuturiActive – împrumuturi neînchise (DataSfarsit IS NULL)
  • vVenituriLunare – total venituri pe lună (din Plati)
  • vTopClienti – top 10 clienți după numărul de împrumuturi

7. Proceduri stocate (CRUD) – descriere

  • spClienti_Create/ReadById/Update/Delete – CRUD pentru clienți
  • spBiciclete_Create / spBiciclete_UpdateStatus / spBiciclete_MoveToStatie – operații biciclete
  • spImprumut_Start – pornește împrumut (validează bicicleta disponibilă, setează status)
  • spImprumut_End – închide împrumut (setează DataSfarsit, status bicicleta + stație returnare)
  • spPlati_Create – înregistrează o plată pentru un împrumut

8. Triggere

8.1 Triggere DML

  • tr_Imprumuturi_PreventDoubleActive – nu permite mai mult de un împrumut activ pe aceeași bicicletă
  • tr_Imprumuturi_CloseSetsBikeAvailable – dacă DataSfarsit se completează manual, bicicleta devine disponibilă

8.2 Trigger DDL

  • tr_DDL_Audit_Bicicleta – loghează evenimente CREATE/ALTER/DROP pentru TABLE/VIEW/PROCEDURE în Error Log

9. Cursor

Procedura spRaport_DisponibilitatePeStatii_Cursor folosește un cursor pentru a parcurge stațiile și a afișa prin PRINT câte biciclete disponibile există în fiecare stație.

10. Utilizatori, roluri și drepturi

Se folosesc roluri de bază de date (SQL Server DB roles) pentru a limita accesul pe obiecte. Acestea sunt separate de tabelul aplicației Roluri.

Rol DB Drepturi (rezumat) Utilizare
db_admin_app SELECT/INSERT/UPDATE/DELETE pe schema dbo + EXECUTE pe proceduri Administrare completă în cadrul DB
db_operator SELECT/INSERT/UPDATE pe Biciclete, Statii, Imprumuturi + EXECUTE Start/End Operator stație / mentenanță / operațional
db_cashier INSERT/SELECT pe Plati + SELECT pe view venituri + EXECUTE spPlati_Create Casierie / financiar
db_readonly SELECT pe schema dbo Raportare / consultare

11. Job-uri Transact-SQL

Job-ul JOB_Biciclete_ReconcilereStatus rulează zilnic și corectează inconsistențe: biciclete marcate 'inchiriata' fără un împrumut activ sunt setate automat 'disponibila'.

Necesită SQL Server Agent pornit și permisiuni în baza msdb.

12. Strategii backup / restore

12.1 Strategie recomandată (simplă)

  • Full backup zilnic (ex. 02:00)
  • Differential backup la 4–6 ore (opțional)
  • Transaction log backup (doar dacă DB este în FULL recovery)

12.2 Scripturi exemplu

-- FULL BACKUP
BACKUP DATABASE bicicleta
TO DISK = 'C:\Backup\bicicleta_full.bak'
WITH INIT, COMPRESSION, STATS = 10;

-- DIFFERENTIAL (opțional)
BACKUP DATABASE bicicleta
TO DISK = 'C:\Backup\bicicleta_diff.bak'
WITH DIFFERENTIAL, INIT, COMPRESSION, STATS = 10;

-- RESTORE (exemplu)
-- RESTORE DATABASE bicicleta
-- FROM DISK = 'C:\Backup\bicicleta_full.bak'
-- WITH REPLACE, RECOVERY, STATS = 10;

În practică, locațiile de backup depind de server și permisiuni; pentru proiect este suficient să explici strategia și să arăți scriptul.

13. Anexe

13.1 Observații

  • Statusurile bicicletelor sunt validate prin CHECK.
  • Împrumuturile active au DataSfarsit NULL.
  • Consistența “o bicicletă nu poate avea două împrumuturi active” este impusă prin trigger DML.
  • Rapoartele sunt implementate prin views pentru a simplifica interogările.
